# AGENTS.md

## 项目目标

本项目基于 Pi Coding Agent 构建个人专属 Agent，包括扩展、工具、命令、技能、提示词及相关配置。

用户不负责 TypeScript 开发。你必须主动完成分析、设计、实现、重构、验证和文档更新，不得把代码修改工作转交给用户。

本项目将被用于 `pi` 的配置目录, 本文件夹即会作为 `~/.pi`

## 首要原则

### 1. 先查证，后编码

涉及 Pi 的接口、事件、类型、目录或配置时，禁止凭记忆实现。

按以下优先级确认事实：

1. 当前项目已安装依赖中的类型声明与源码；
2. Pi 官方最新文档；
3. Pi 官方仓库中的源码和示例。

必须确认项目实际使用的包名和版本，禁止盲目照搬旧版示例。无法确认的接口不得虚构，应继续查阅源码或类型定义。

### 2. 保持最小实现

* 只编写完成当前需求所必需的代码。
* 禁止预留功能、无需求抽象、重复封装、无意义兼容层和过度防御性代码。
* 优先使用 Pi 官方已有能力，不重复实现框架功能。
* 能删除旧实现时，不新增适配层绕过问题。

### 3. 允许破坏性重构

* 无需保持旧接口、旧配置或旧目录结构的兼容性。
* 现有设计妨碍简洁性时，直接重构并同步修改所有调用方。
* 删除废弃代码、无效配置、未使用依赖和过时注释。
* 不保留“以后可能使用”的代码。

## Pi 功能选型

始终选择满足需求的最简单官方机制：

* 项目长期开发规范：`AGENTS.md`
* 追加默认系统提示词：`~/.pi/agent/APPEND_SYSTEM.md`
* 完全替换系统提示词：`~/.pi/agent/SYSTEM.md`
* 固定任务模板：`~/.pi/agent/prompts/`
* 按需知识或复杂工作流：`~/.pi/agent/skills/`
* 工具、命令、事件钩子或运行时行为：`~/.pi/agent/extensions/`

Markdown 配置能够解决的问题，不得使用 TypeScript 扩展实现。

## 提示词设计

系统提示词、工具提示词、模板提示词、技能说明及其他面向模型的文本，必须以**最少 token 获得最佳效果**为设计目标。

* 只保留会实际影响模型行为的信息。
* 使用简短、明确、可执行的指令。
* 删除寒暄、背景铺垫、重复强调、同义复述和无效示例。
* 已由上层提示词明确的规则，不在下层重复。
* 能用一条通用规则覆盖的内容，不枚举多个相似规则。
* 优先使用肯定式指令，减少冗长的反向约束。
* 工具名称、描述和参数说明必须简短、精确、无歧义。
* 不向模型解释实现细节，除非这些细节会影响正确决策。
* 示例仅在没有示例容易产生歧义时使用，并保持最小数量。
* 不通过堆叠提示词修复代码、接口或架构问题。
* 新增提示词前，先检查是否可以删除、合并或改写现有内容。
* 修改完成后，必须检查提示词是否存在重复、冲突或可进一步压缩的内容。

在不降低准确性、稳定性和可维护性的前提下，始终选择 token 更少的表达。

## 代码要求

* 使用严格、清晰并符合项目配置的 TypeScript。
* 优先使用精确类型和类型推导。
* 禁止随意使用 `any`、非空断言或双重类型断言。
* 函数保持单一职责，模块边界直接反映业务职责。
* 避免无意义的类、管理器、工厂、包装器和仅调用一次的辅助层。
* 不为假设中的未来需求设计扩展点。
* 标准库或现有依赖能够完成时，不增加新依赖。
* 错误必须在合适的边界处理，禁止静默吞掉异常。
* 异步操作必须正确处理取消、资源释放和并发写入问题。
* 面向模型的返回内容必须结构清晰、信息充分且尽量简短。

## 中文注释

* 所有导出接口、工具、命令、配置项、关键状态和非显然逻辑必须有准确的中文注释。
* 注释应解释设计意图、约束、边界条件或 Pi 接口行为。
* 禁止逐行翻译代码、复述变量名或编写无信息量注释。
* 修改行为时，必须同步更新相关注释和文档。
* 能通过清晰命名和简单结构表达的内容，不添加冗余注释。

## 工作方式

实施前：

1. 阅读项目结构、`package.json`、TypeScript 配置、Pi 配置及相关源码。
2. 查询任务涉及的 Pi 官方接口和本地类型定义。
3. 判断现有实现应复用、重构还是删除。
4. 选择改动范围最小、结构最清晰的方案。

实施中：

* 完整修改所有相关调用点、类型、配置、测试和文档。
* 不使用临时补丁掩盖结构问题。
* 不擅自修改与任务无关的行为。
* 发现当前任务范围内存在明显冗余时，直接简化。
* 提示词与代码同时变更时，优先从架构上解决问题，避免用额外提示词补偿。

实施后：

1. 检查最终差异，删除调试代码和无用内容。
2. 检查所有新增提示词的必要性、重复性和 token 成本。
3. 根据项目已有脚本运行格式化、静态检查、类型检查和测试。
4. 对 Pi 扩展执行适当的加载或冒烟验证。
5. 确认没有未使用代码、失效引用、重复实现或遗漏的文档更新。

## 沟通要求

* 默认使用中文说明。
* 不要求用户阅读或修改 TypeScript 代码。
* 对普通技术细节自行作出合理决定，不反复询问用户。
* 仅当产品行为存在实质歧义、可能造成数据损失，或缺少无法从仓库和官方资料获得的信息时，才提出问题。
* 完成后简要说明：

  * 实现内容；
  * 关键设计决定；
  * 删除或破坏性修改的内容；
  * 执行的验证及结果；
  * 尚未验证的事项。

## 完成标准

只有同时满足以下条件，任务才算完成：

* 行为符合需求；
* Pi 接口已通过官方资料或本地类型确认；
* 实现是当前需求下足够简单的方案；
* 提示词不存在明显冗余，并已尽量降低 token 成本；
* 类型检查和相关测试通过；
* 中文注释与文档准确；
* 不存在为兼容旧实现而保留的多余代码。
